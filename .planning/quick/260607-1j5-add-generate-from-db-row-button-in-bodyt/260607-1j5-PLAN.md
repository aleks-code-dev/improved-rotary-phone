## Context

The `BodyTab` in the request editor (renderer) already exposes a `Generate from DTO` button (`BodyTab.tsx:144-156`) that fills the JSON body from the endpoint's detected DTO schema via `window.api.body.generateDto`. Separately, the `DatabasePanel` shows a `DbRowDetail` component with a `Use this row → body` button (`DbRowDetail.tsx:99-117`) that pulls a real row from the connected DB and maps it onto the DTO via `window.api.db.mapRowToDto`, then writes the result into the body via `setBody`.

There is no shortcut for the common workflow "I just clicked a row in the DatabasePanel and now I want to use it as the body of the currently active request." The user has to leave the row in place and tab over to the body editor. The fix is to expose the same `mapRowToDto` mechanism from inside the body editor itself, right next to `Generate from DTO`. The user-confirmed scope is strictly renderer-only: reuse the existing IPC, no new channels, no main/helper/preload changes, no new dependencies, no edits to `DbRowDetail.tsx` (its existing `Use this row → body` button must keep working).

The state-sharing problem: `selectedConnectionId` and `selectedRow` currently live as local `useState` in `DatabasePanel.tsx:33-34`. `BodyTab` is a sibling component, not a child, so it cannot read those values. The chosen fix is a small Zustand store (`src/renderer/store/dbSelection.ts`) that mirrors the same patterns as the existing `useRequest` / `useTabs` / `useEndpointsStore` stores. `DatabasePanel` keeps its local `useState` for any local reads but also writes to the new store at the two existing write sites (`setSelectedConnectionId` at the connection-list `onClick`, `handleRowSelect` at the row-click handler). `BodyTab` reads from the new store. This keeps the diff small — local reads in `DatabasePanel` stay valid, the store is the cross-component source of truth.

## Approach

**Task 1 — Add the cross-component store.** Create `src/renderer/store/dbSelection.ts` with the `DbSelectionState` interface and the `useDbSelection` Zustand hook specified in the task brief. Shape: `{ selectedConnectionId, selectedTableName, selectedRow, setSelection, clear }`. The selected-row payload is `{ row, schema }` to keep the same shape the panel already uses; `selectedTableName` is the bare table name (no `schema.` prefix) because that is exactly what `db.mapRowToDto` expects. In `DatabasePanel.tsx`, at the two write sites only, mirror the local state update with a `useDbSelection.getState().setSelection({...})` call. Do NOT remove the local `useState` — local reads in `DatabasePanel` still depend on it. Do NOT write to the store from anywhere else (no clear on disconnect, no sync on tab change in this quick).

**Task 2 — Add the button and its handler in `BodyTab`.** Add a new button labeled `Generate from DB row` immediately after the existing `Generate from DTO` button inside the same `{spec?.detectedDto && body.contentType === 'application/json' && (<>...</>)}` block. The button has its own `isGeneratingRow` local state and its own `rowError` local state — do NOT share state with the existing DTO generation. Read `selectedConnectionId`, `selectedTableName`, `selectedRow` from `useDbSelection`. Visibility is the conjunction of (a) the existing DTO block precondition, (b) `spec.detectedDto?.fqn` truthy, and (c) all three of the selected-row fields non-null. The click handler is structurally identical to `DbRowDetail.handleUseRow` (`DbRowDetail.tsx:13-26`) but with full error capture (not silent `catch`) and an inline red error label to the right of the button. The success path is identical to the existing DTO button: `setBody(tabId, { mode: 'raw', contentType: 'application/json', text: result.bodyJson })`.

Why not wire through the existing `body:auto-filled` event that `DatabasePanel.handleUseRow` already fires? That event switches the active tab in the request editor; the user is *already* on the Body tab when they click the new button, and the new button lives *inside* that tab, so the tab-switch is unnecessary. Bypass the event.

### Task 1: Add `useDbSelection` store and wire `DatabasePanel` writes

**Files:**
- `src/renderer/store/dbSelection.ts` (new)
- `src/renderer/components/Database/DatabasePanel.tsx`

**Action:**

In `src/renderer/store/dbSelection.ts` (new file), create the store following the same pattern as `src/renderer/store/endpoints.ts` and `src/renderer/state/useTabs.ts`:

```ts
import { create } from 'zustand';

export interface DbSelectedRow {
  row: Record<string, unknown>;
  schema: string | null;
}

export interface DbSelectionState {
  selectedConnectionId: string | null;
  selectedTableName: string | null;
  selectedRow: DbSelectedRow | null;
  setSelection: (sel: Partial<Pick<DbSelectionState, 'selectedConnectionId' | 'selectedTableName' | 'selectedRow'>>) => void;
  clear: () => void;
}

export const useDbSelection = create<DbSelectionState>((set) => ({
  selectedConnectionId: null,
  selectedTableName: null,
  selectedRow: null,
  setSelection: (sel) => set((s) => ({ ...s, ...sel })),
  clear: () => set({ selectedConnectionId: null, selectedTableName: null, selectedRow: null }),
}));
```

(If `src/renderer/store/` does not already contain an `endpoints.ts`, fall back to mirroring the structure of `useRequest.ts` exactly. Confirm with `ls src/renderer/store/` before writing.)

### Task 2: Add the 'Generate from DB row' button and its handler in BodyTab

**Files:**
- `src/renderer/components/RequestEditor/BodyTab.tsx`

**Action:**

In `BodyTab.tsx`:

1. Add the import at the top with the other store imports:
   ```ts
   import { useDbSelection } from '../../store/dbSelection';
   ```

2. Add three selector reads inside the component body, next to the existing `useRequest` reads at line 33-36:
   ```ts
   const selectedConnectionId = useDbSelection((s) => s.selectedConnectionId);
   const selectedTableName = useDbSelection((s) => s.selectedTableName);
   const selectedRow = useDbSelection((s) => s.selectedRow);
   ```

3. Add two new local states next to the existing DTO-generation state block (line 64-67):
   ```ts
   const [isGeneratingRow, setIsGeneratingRow] = useState(false);
   const [rowError, setRowError] = useState<string | null>(null);
   ```

4. Add a new handler near `handleGenerateDto` (line 87-108), structurally identical to `DbRowDetail.handleUseRow` (`DbRowDetail.tsx:13-26`) but with full error capture:
   ```ts
   const handleGenerateFromRow = useCallback(async () => {
     if (!spec?.detectedDto?.fqn || !selectedConnectionId || !selectedTableName || !selectedRow || isGeneratingRow) return;
     setIsGeneratingRow(true);
     setRowError(null);
     try {
       const result = await window.api.db.mapRowToDto({
         connectionId: selectedConnectionId,
         tableName: selectedTableName,
         rowId: selectedRow.row,
         dtoFqn: spec.detectedDto.fqn,
       });
       if (result.ok) {
         setBody(tabId, { mode: 'raw', contentType: 'application/json', text: result.bodyJson });
       } else {
         setRowError((result as any)?.error ?? 'Failed to generate from row');
       }
     } catch (err: any) {
       setRowError(err?.message ?? 'Failed to generate from row');
     } finally {
       setIsGeneratingRow(false);
     }
   }, [spec, selectedConnectionId, selectedTableName, selectedRow, isGeneratingRow, tabId, setBody]);
   ```

5. Inside the existing DTO button block (line 130-158, the `{spec?.detectedDto && body.contentType === 'application/json' && (<>...</>)}` block), immediately after the `Generate from DTO` button (after line 156's closing `</button>`), add a new button and its inline error label. The visibility guard is the conjunction of the outer block's precondition plus `selectedConnectionId && selectedTableName && selectedRow`:
   ```tsx
   {selectedConnectionId && selectedTableName && selectedRow && (
     <>
       <button
         onClick={handleGenerateFromRow}
         disabled={!helperOnline || isGeneratingRow}
         title={'Use selected row from ' + selectedTableName + ' as body'}
         style={{
           ...formatBtnStyle,
           color: 'var(--color-accent)',
           borderColor: 'var(--color-accent)',
           fontWeight: 600,
         }}
       >
         {isGeneratingRow ? 'Generating…' : 'Generate from DB row'}
       </button>
       {rowError && (
         <span style={{ color: 'var(--ds-method-delete)', fontSize: 11, marginLeft: 'var(--space-2)' }}>
           {rowError}
         </span>
       )}
     </>
   )}
   ```

Notes on the implementation choices:
- Reusing the same `formatBtnStyle` + accent color scheme keeps the two buttons visually consistent. They are siblings of each other inside the same flex row.
- The error span sits inline to the right of the button (no banner, no separate row). It uses `var(--ds-method-delete)` (the same red the prior quick 260607-0in used for the "Not connected" error in `DbTableTree.tsx`). It uses `var(--ds-*` tokens for the inline elements — these are the design-system tokens the panel already uses; the rest of `BodyTab` uses `var(--color-*` tokens (older namespace). Either works because both namespaces exist in the global CSS. Stick with `var(--ds-method-delete)` for the error to match the recent prior work; the button itself keeps the existing `var(--color-accent)` styling to match the sibling DTO button.
- The handler does NOT clear `rowError` when the body is successfully written. (The error is gone the moment the button is clicked again via `setRowError(null)` at the top of the next invocation, and clearing on success creates a flash. Mirror the silent-error handling of the DTO button which never sets an error state at all.)
- The `setBody` call uses the same body shape the DTO button uses and the same shape `DbRowDetail.handleUseRow`'s caller (`DatabasePanel.handleUseRow` at line 74-83) uses: `{ mode: 'raw', contentType: 'application/json', text: result.bodyJson }`. Confirmed three independent call sites agree.
- `isGeneratingRow` is intentionally separate from `isGenerating`. The two buttons can be clicked independently; their loading spinners are independent.

## Risks

- **Stale `selectedRow` across connection switches.** When the user switches the active connection in `DatabasePanel`, the existing code already calls `setSelectedRow(null)` (line 148). The Task 1 wiring also clears the store's `selectedTableName` and `selectedRow` (and sets the new `selectedConnectionId`) at the same site, so the new button correctly disappears at the right moment. If the user switches connections *without* the prior write to `setSelectedRow(null)` happening for any reason (e.g., a future refactor), the button would render using a row from the old connection. The Task 1 changes make the store writes mirror the local-state writes at the *same* call sites, so this is a non-issue today; future refactors must keep them in sync.
- **The `mapRowToDto` IPC contract.** `DbRowDetail.tsx:16-21` calls `window.api.db.mapRowToDto({ connectionId, tableName, rowId: row, dtoFqn })` and treats the return shape as `{ ok, bodyJson }` on success. The new button in Task 2 uses the exact same call and treats the return shape the same way. The fallback branch (`else { setRowError((result as any)?.error ?? 'Failed to generate from row') }`) handles the `{ ok: false, error }` shape defensively; if the actual shape is different in the unhappy case, the error message will say `'Failed to generate from row'` which is acceptable. Confirmed by reading `DbRowDetail.tsx:16-25` — it does NOT differentiate `result.ok === false` from a thrown error; the new button is strictly more defensive.
- **Helper is offline.** Both buttons share the `helperOnline` flag and the `disabled={!helperOnline || ...}` pattern. The new button does not introduce a new offline state.
- **No tests for the new store or button.** The codebase has existing renderer tests but no component-level tests for `BodyTab` or `DatabasePanel` (the most recent quick 260607-0in also relied on `npm test` and manual verification). The `Verify` step relies on `npm test` to confirm the existing 82 tests still pass; new behavior is verified manually per the same convention.
- **No changes to the renderer TypeScript surface for `window.api.db.mapRowToDto`.** `src/preload/index.d.ts` does not declare a `db` member at all (prior quick 260607-0in flagged this; it is a pre-existing issue with the implicit-any in the renderer build, not introduced here). The new code uses the same un-typed call site as `DbRowDetail.tsx:16`.
- **Re-render churn.** The new `useDbSelection` selector reads return stable references when the store is unchanged (Zustand returns the same value), so the parent `BodyTab` does not re-render on every store update. The three independent selectors mean the component only re-renders when one of the three read fields actually changes. The local `isGeneratingRow` and `rowError` states are component-local and isolated from the existing DTO-generation state.
