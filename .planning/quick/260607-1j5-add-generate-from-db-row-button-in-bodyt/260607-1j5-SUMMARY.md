## Decisions Made

- **Mirror the local `useState` writes at the same call sites** — The plan called for the smallest possible diff: insert the `useDbSelection.getState().setSelection({...})` call next to each existing local write (not as a replacement). This keeps the local state as the source of truth inside `DatabasePanel` and makes the store a pure cross-component broadcast layer.
- **Pass `rowId: selectedRow.row` (the whole row)** — Matches the existing `DbRowDetail.handleUseRow` call signature (`rowId: row` where `row` is `Record<string, unknown>`). Helper-side uses the row as the primary-key lookup vector; passing the full row preserves the existing call-site contract with zero risk.
- **Button gated on the conjunction, not a separate visible/invisible state** — Outer `{spec?.detectedDto && body.contentType === 'application/json' && ...}` is the existing DTO block. Inside it, the new fragment is `{selectedConnectionId && selectedTableName && selectedRow && <>...</>}`. Both must be true for the button to render. This matches the plan's "no-row-path" and "no-DTO-path" verification cases exactly.
- **Inline error span, not a banner** — The plan cites `--ds-method-delete` (the red token used in `DbTableTree`'s Load Tables error) and a small font (11) with `marginLeft: var(--space-2)`. Banner was explicitly rejected by the plan.
- **`isGeneratingRow` and `rowError` are independent of `isGenerating` and `cycleRefs`** — The plan's notes call this out: "The two buttons can be clicked independently; their loading spinners are independent." The DTO button's behavior is unchanged.
- **Do NOT clear `rowError` on success** — The plan's notes are explicit: "clearing on success creates a flash." `rowError` is cleared at the top of the next click via `setRowError(null)`, which is the same lifecycle as the silent-catch of the DTO button (no error state at all on the DTO side).
- **Skip the `body:auto-filled` event** — The plan calls this out as a deliberate choice: the new button lives inside the body tab, so dispatching an event that switches the active tab would be a no-op flicker. The new code calls `setBody` directly.
- **`result.ok === false` branch uses `(result as any)?.error ?? 'Failed to generate from row'`** — `DbRowDetail.handleUseRow` does not differentiate `result.ok === false` from a thrown error; the new code is strictly more defensive, falling back to a generic message if the error shape is unknown.
