## Context

Phase 8 (Database Integration) shipped a `DatabasePanel` rendered as a fixed-width 340px right-pane tool window in `App.tsx` (line 114). Three regressions reported in quick 260607-0in:

**Symptom 1 — Panel is not resizeable.** `DatabasePanel.tsx:86` hard-codes `width: 340`. There is no drag handle on the left edge. The codebase already has a reusable `<Splitter orientation="vertical" onResize={...} />` component (`src/renderer/components/Splitter.tsx`) used for vertical dividers, but it is not currently imported anywhere — it was prepared for exactly this use case. Fix: lift width into `App.tsx` state, render the Splitter between the main pane and the panel, and feed the width through as a prop.

**Symptom 2 — "Load Tables" does nothing.** `DbTableTree.tsx:58-64` `loadTables` calls `window.api.db.listTables({ connectionId })` which is wired correctly. The button itself is fine. The real cause is the missing upstream wiring described in Symptom 3 — the helper has no HikariCP pool for that `connId`, so the helper returns `error: "Not connected"`, the main process `db:listTables` handler swallows it (`router.ts:600-603`) and returns `[]`, then `setTables([])` renders the "No tables found" placeholder. Compounding the UX failure: `loadTables` has no `setLoading` indicator and its `catch` block is `/* ignore */` (line 63) — the user gets no feedback that anything happened.

**Symptom 3 — `db:listTables failed { error: 'Not connected' }` in console.** End-to-end trace:

- Renderer: `DbConnectionForm.handleSave` (`DbConnectionForm.tsx:100-112`) calls `window.api.db.connections.create({ name, url, user, password, dbType })` and stops. **It never calls `window.api.db.connect({ connectionId })`**.
- Main: `db:connect` handler (`router.ts:544-561`) is fully implemented and would register the HikariCP pool via `dbConnectionsService.readConnectionDecrypted(connectionId)` + `client.request('db:connect', { connId, url, user, password })`. It is just never invoked from the renderer Save path.
- Helper: `HelperJsonRpcServer.java:132-149` handles `db:connect` by calling `dbConnectionManager.connect(connId, url, user, password)` which adds the `HikariDataSource` to `Map<String, HikariDataSource> pools` (`DbConnectionManager.java:11-20`).
- Helper: `db:listTables` (line 184-206) looks up the pool by `connId`. If the pool is missing it returns `error.message = "Not connected"`. The renderer never registered the connId, so the lookup misses.

The `db:testConnection` (ad-hoc, URL/user/password) succeeds because it creates a throwaway pool in `DbConnectionManager.testConnection` (line 29-36). It does **not** register a pool. So the user's flow — Test (passes) → Save → select connection → Load Tables — ends with the helper never having a pool for the saved connId.

The same problem also blocks `fetchRows` and `mapRowToDto` for the same reason, but the user only reported `listTables` because it's the first DB call after Save.

**Solution shape (renderer-only):** Two atomic changes — one to the JSX in `App.tsx`/`DatabasePanel.tsx` to add resize; one to the connection lifecycle in `DbConnectionForm.tsx`/`DatabasePanel.tsx`/`DbTableTree.tsx` to wire Save→connect, select→auto-connect, and surface Load Tables errors with a loading indicator. No main-process, Java helper, IPC, or storage changes. The `db:connect` IPC channel and its handler are already complete and correct.

## Approach

**Task 1 — Resize handle.** In `App.tsx`, introduce `const [dbPanelWidth, setDbPanelWidth] = useState<number>(() => Number(localStorage.getItem('pc.dbPanelWidth')) || 340)`. Clamp the setter to `[240, 700]`. Persist to `localStorage` on change. Render `<Splitter orientation="vertical" onResize={(d) => setDbPanelWidth(w => clamp(w - d, 240, 700))} />` immediately before the `{dbPanelOpen && <DatabasePanel ... />}` fragment. Pass `width={dbPanelWidth}` to `DatabasePanel`. In `DatabasePanel.tsx`, accept a `width: number` prop and replace the inline `width: 340` with `width` from props. The Splitter already exists in the codebase; it is unused (`grep Splitter src/renderer` shows only the component file itself).

**Task 2 — Wire the connection lifecycle and surface Load Tables state.**

1. **`DbConnectionForm.handleSave`**: after `connections.create` returns `{ id }`, immediately call `window.api.db.connect({ connectionId: id })`. Wrap in try/catch. If `connect` returns `{ ok: false }` or throws, surface the error to the user (use the same `testResult` UI pattern: red text "✗ Could not connect: …") and keep the form open. On success, call `onConnectionCreated(connectionId)` so the parent can mark the connection as connected and skip auto-connect on select.
2. **`DatabasePanel`**: change `handleConnectionCreated` to accept the new connection's `id`, and after `refreshConnections` set `setConnections(prev => prev.map(c => c.id === id ? { ...c, connected: true } : c))` (or, if connect failed, leave `connected: false`). On selecting a connection from the list, if `!conn.connected`, fire `db.connect({ connectionId: conn.id })` in the background and on success update the dot. This makes the panel robust to users who skip Test and hit Save, or who restart the app and re-open the panel (helper pools are in-memory; only Saved-on-disk + auto-reconnect works).
3. **`DbTableTree.loadTables`**: add a `tablesLoading` state, set it true before the call and false in `finally`. Capture the error in a `tablesError` state and render it above the "No tables found" placeholder when set. Remove the silent `catch { /* ignore */ }`.

The "connected" dot in the connections list (`DatabasePanel.tsx:167-173`) already reads from `conn.connected`, so updating the in-memory state propagates automatically.

## Risks

- **Pool leak on re-Save.** If the user edits and re-Saves a connection with a new password, the new `db.connect` call closes the old pool (`DbConnectionManager.connect` line 15-17) and replaces it. No leak.
- **Pool leak on connection delete.** `DatabasePanel` does not currently call `db.disconnect` on delete. The connection file is removed from disk (`db-connections.ts:80-85`) but the HikariCP pool stays in the helper until JVM restart. Pre-existing minor leak; out of scope for this quick.
- **Helper restart loses pools.** When the user restarts the helper (or it crashes and respawns), all HikariCP pools are lost. The new auto-connect-on-select handles this for the active selection; a user who hasn't touched the panel since restart will see gray dots that turn green on click. Acceptable for v3.0; full re-hydration on bootstrap is a v3.1 improvement.
- **localStorage key collision.** The key `pc.dbPanelWidth` is namespaced (`pc.` prefix); no other code reads or writes this key. Confirmed by `grep localStorage src/renderer` returning zero hits — we are introducing the first usage.
- **The `db:listTables` catch change in `router.ts` could break consumers.** Currently zero other callers depend on the silent `[]` return — `grep listTables src/` shows only `DatabasePanel.tsx`'s `DbTableTree` consumer. Safe.
- **Stale `index.d.ts`.** `src/preload/index.d.ts` does not declare a `db` member at all (it ends at `project:`), yet `window.api.db.*` is used throughout the renderer. The codebase must be relying on implicit `any` / non-strict TS in the renderer build, or the .d.ts is not the one being consumed. Pre-existing issue, not introduced here.
