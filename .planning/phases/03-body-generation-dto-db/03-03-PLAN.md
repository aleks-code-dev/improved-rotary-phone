1. Create `ColumnFieldNameMatcher.java` (per RESEARCH.md §Don't Hand-Roll — snake_case↔camelCase):
   - `public static Map<String, String> autoMap(List<String> dbColumns, List<String> dtoFields)` — returns map of dbColumn → dtoField
   - Algorithm: for each DTO field, check if any DB column matches case-insensitively after normalizing snake_case↔camelCase
   - Transform: `user_name` → `userName` (snake_to_camel), `userName` → `user_name` (camel_to_snake)
   - Both directions checked; exact case-insensitive match takes priority over transform match
   - Edge cases: consecutive underscores, leading underscore, underscore before digit

2. Create `RowToJsonMapper.java` (per RESEARCH.md §Pattern 5 + §Code Examples):
   - Constructor takes: `TypeNormalizer normalizer` (injected per driver)
   - `public String mapRow(HikariDataSource pool, String tableName, Map<String, Object> rowId, String dtoFqn, Map<String, String> columnMapping)`:
     a. Walk the DTO via DtoWalker to get expected shape (field names + types)
     b. Fetch the specific row: `SELECT * FROM tableName WHERE pk = ?` (use primary key from rowId)
     c. For each column in ResultSet: apply TypeNormalizer, then map to DTO field via columnMapping (or auto-map via ColumnFieldNameMatcher)
     d. Build ObjectNode matching DTO shape — only include fields that have a matching column
     e. Extra DB columns not in DTO are excluded
     f. Unmapped required DTO fields are tracked and returned as warnings
   - Returns JSON string (the body) + mapping info + warnings
   - `public String fetchRows(HikariDataSource pool, String tableName, String schema, String mode, String idValue, String whereClause, int limit)`:
     a. Build SQL based on mode:
        - `firstN`: `SELECT * FROM tableName LIMIT limit`
        - `byId`: `SELECT * FROM tableName WHERE id = ?` (parameterized)
        - `byWhere`: `SELECT * FROM tableName WHERE ${whereClause} LIMIT limit` (validate: no semicolons, no DROP/DELETE/UPDATE/INSERT — only SELECT)
     b. Use `setFetchSize(50)` for PostgreSQL
     c. Cap at LIMIT 100 (RESEARCH.md §Pitfall 7)
     d. Return JSON with: rows array, columns metadata (name, typeName, nullable), truncated flag, totalCount
   - SQL injection mitigation (per RESEARCH.md §Pitfall 7): for `byId` mode use PreparedStatement. For `byWhere` mode: validate no destructive keywords, wrap in PreparedStatement if possible, reject if contains `;`

3. Extend `HelperJsonRpcServer.java` dispatch loop:
   - `"db:fetchRows"` → extracts connectionId, tableName, schema, mode, idValue, whereClause, limit; gets pool from DbConnectionManager; calls `rowToJsonMapper.fetchRows()`; returns result
   - `"db:mapRowToDto"` → extracts connectionId, tableName, rowId, dtoFqn, columnMapping; gets pool; calls `rowToJsonMapper.mapRow()`; returns `{ bodyJson, mapping, coverage, warnings }`
   - All methods: try/catch, NEVER log connection details or passwords

4. Add new IPC schemas to `channels.ts` (only the `db:mapRowToDto` schemas — `db:fetchRows` was added in 03-02):
   - `DbMapRowToDtoArgsSchema`: `{ connectionId: z.string().uuid(), tableName: z.string(), rowId: z.record(z.string(), z.unknown()), dtoFqn: z.string(), columnMapping: z.record(z.string(), z.string()).default({}) }`
   - `DbMapRowToDtoResultSchema`: `{ ok: z.boolean(), bodyJson: z.string(), mapping: z.array(z.object({ column: z.string(), field: z.string(), compatibility: z.enum(['exact', 'compatible', 'incompatible']) })), coverage: z.object({ mapped: z.number(), required: z.number(), total: z.number() }), warnings: z.array(z.object({ code: z.string(), message: z.string() })).default([]) }`
   - Export inferred types

5. Register `db:mapRowToDto` IPC handler in `router.ts`:
   - Parse args, get connection (decrypt password), relay to helper
   - Return validated result

6. Add `db:mapRowToDto` to preload `index.ts` under `db` section

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer→main | IPC payloads cross process boundary; Zod validates at handler entry |
| main→helper | DB row data flows from helper to main to renderer; no credentials in row data |
| helper→database | SQL queries constructed from user input (WHERE clause); PreparedStatement required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-09 | Tampering | SQL injection via WHERE clause | mitigate | Validate WHERE: no semicolons, no destructive keywords (DROP/DELETE/UPDATE/INSERT), use PreparedStatement for byId mode |
| T-03-10 | Information Disclosure | DB row data in renderer memory | mitigate | Row data is transient UI state; not persisted to disk; cleared on connection close |
| T-03-11 | Denial of Service | Large result set freezes UI | mitigate | LIMIT 100 cap, setFetchSize(50), virtualized list via react-virtuoso for large row sets |
| T-03-12 | Tampering | Column mapping override | mitigate | Mapping is user-initiated; auto-map is default; explicit overrides take precedence |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages; all existing from Phase 1 |
</threat_model>
