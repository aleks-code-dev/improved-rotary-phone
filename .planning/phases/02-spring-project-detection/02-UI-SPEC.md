## Visual Hierarchy & Focal Points

**Primary focal point:** EndpointsTree sidebar (after scan) / "Select Spring Project" CTA button (empty state). The endpoint tree is the main new UI surface — users scan it to discover available endpoints.

**Secondary:** DTO class panel (when endpoint selected) — provides context about the request body class below the endpoint tree.

**Tertiary:** Status bar scanner indicator — ambient awareness of scan state without competing for attention.

**Visual entry sequence:** 
1. Sidebar toggle label "ENDPOINTS" draws the eye to the new section
2. CTA button or endpoint tree content fills the section
3. Status bar dot provides background reassurance about scanner health

### Accent Reserved For (explicit whitelist)

The `--color-accent` token is reserved exclusively for:
1. **Primary action buttons** — "Select Spring Project", "Rescan", "Create" in modals
2. **Focus rings** — `outline: 2px solid var(--color-accent)` on focused inputs/buttons
3. **Progress bar fill** — scan progress indicator fill color
4. **Scan status dot** — ● green scanner-healthy indicator in status bar (uses `--color-status-2xx`, NOT accent)
5. **Active tab indicator** — bottom border on active editor tab

Never use accent for: decoration, backgrounds (use `--color-bg` / `--color-bg-elevated`), non-primary text (use `--color-fg` / `--color-fg-muted`), or neutral UI chrome.

## Interaction Contracts

### 1. Project Selection

**Trigger:** "Select Spring Project" button clicked in sidebar Endpoints section.

**Flow:**
1. Native folder picker dialog opens (IPC `dialog.showOpenDialog` with `properties: ['openDirectory']`)
2. User selects a folder → path sent to main process
3. Main process persists the path to `settings.json` under `lastSpringProjectPath`
4. Main process invokes JVM helper: `scanner.scan({ projectRoot: path })`
5. UI transitions to scanning state (see §2)

**Visual:**
```
┌────────────────────────────────────┐
│ ENDPOINTS                          │
│                                    │
│ ┌────────────────────────────────┐ │
│ │  Select Spring Project         │ │  ← Primary CTA, accent bg, white text
│ └────────────────────────────────┘ │
└────────────────────────────────────┘
```
Button style: `background: var(--color-accent)`, `color: white`, `border: none`, `border-radius: var(--radius-1)`, `padding: var(--space-1) var(--space-3)`, `font-size: 12px`, `font-weight: 600`, `cursor: pointer`.

### 2. Scan Progress

**Triggers:** After project selection, on app re-open (if valid last project path), on manual "Rescan".

**Visual:**
```
┌────────────────────────────────────┐
│ ENDPOINTS                          │
│ ● Scanning /Users/me/spring-app... │  ← Project path, 11px muted, truncated
│ ████████░░░░░░░░░░ 40%             │  ← Progress bar, accent fill
│ Found 127 endpoints so far...      │  ← Live counter, 11px muted, updates in real time
└────────────────────────────────────┘
```

**Progress bar specs:**
- Height: 4px
- Background: `var(--color-border)`
- Fill: `var(--color-accent)`
- Border-radius: `var(--radius-1)`
- Width: 100% of sidebar
- Indeterminate mode (animated shimmer) when endpoint count is unknown
- Determinate mode (percentage fill) when file count is known

**Live counter:** Updates via IPC events from main process (`scanner:progress` channel). Format: `Found {N} endpoints so far...` where N increments as each controller file is processed.

### 3. Endpoint Tree (Scan Complete)

**Visual:**
```
┌────────────────────────────────────┐
│ ENDPOINTS                          │
│ /Users/me/spring-app          [⟳] │  ← Project path + Rescan button
│                                    │
│ ▶ UserController (12)              │  ← Controller name, endpoint count
│ ▼ OrderController (8)              │  ← Expanded controller
│     GET   /api/orders              │  ← Endpoint entries
│     POST  /api/orders              │
│     GET   /api/orders/{id}         │
│     PUT   /api/orders/{id}         │
│     DELETE /api/orders/{id}        │
│ ▶ ProductController (5)            │  ← Collapsed controller
│ ▶ AuthController (3)              │
└────────────────────────────────────┘
```

**Tree item specs (matching existing `CollectionsTree` pattern):**
- **Controller row:** 12px font, weight 400 (600 when active), expand/collapse arrow ▶/▼ at 10px, controller name + endpoint count badge
- **Endpoint row:** 12px font, weight 400, indented `--space-5` (24px) from controller, method badge at 10px weight 600 colored via `var(--color-method-{verb})`, path in `ui-monospace` 12px
- **Padding per item:** `var(--space-1) var(--space-2)` (4px 8px)
- **Border-radius:** `var(--radius-1)` (4px)
- **Hover background:** `var(--color-bg-hover)` (to be defined: `oklch(0.92 0.005 250)` light / `oklch(0.25 0.01 250)` dark)
- **Active/selected:** `font-weight: 600`, `background: var(--color-bg-hover)`

**Sort order:** Controllers alphabetically by class name. Endpoints within controller by path, then by HTTP method (GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS).

### 4. Click-to-Prefill Endpoint

**Trigger:** Click on an endpoint row in the EndpointsTree.

**Flow:**
1. Endpoint metadata retrieved from scan cache (method, full path, path vars, query params, headers, consumes, produces, DTO FQN)
2. New tab created via `useTabs.getState().addTab()` with method and URL
3. Request store populated with prefilled data:
   - Method and URL set
   - Headers: `Content-Type` from `consumes`, `Accept` from `produces`
   - Path params: placeholders from path template (e.g., `{id}` → empty editable value)
   - Query params: detected `@RequestParam` parameters
   - Body mode: `raw` + `application/json` if DTO class resolved; `none` otherwise
   - Body content: DTO schema placeholder JSON (e.g., `{}`) — full DTO body generation is Phase 3
4. Tab bar shows method badge + endpoint path as tab title
5. Editor focus moves to request URL bar

**Tab title format:** `{METHOD} {endpoint_path}` (e.g., `POST /api/orders`)

### 5. DTO Class Panel

**Trigger:** Endpoint selected that has a resolved DTO class.

**Visual:**


**Placement:** Appears below the endpoint tree in the sidebar, below EndpointsTree and above the next sidebar section, only when an endpoint with a resolved DTO is selected. Collapsible (expanded by default).

**DTO panel specs:**
- Section header: 11px uppercase, weight 600, `var(--color-fg-muted)`
- FQN: 12px `ui-monospace`, weight 600, `var(--color-fg)`
- Field list: 11px `ui-monospace`, weight 400, `var(--color-fg-muted)`, one per line
- Field names: `var(--color-fg)`, types: `var(--color-fg-muted)`
- Panel background: transparent (inherits sidebar bg)
- Panel padding: `var(--space-2) 0`

**Empty state (no DTO):** When selected endpoint has no body-accepting method (GET, DELETE, HEAD, OPTIONS): show muted text "No DTO class resolved for this endpoint" at 12px.

### 8. App Re-open Rescan

**Behavior:** On app launch, if `lastSpringProjectPath` exists in `settings.json` AND the path still exists on disk:
1. Sidebar shows scanning state immediately
2. JVM helper invoked for scan
3. Scan results populate the tree

If the path no longer exists on disk:
1. Sidebar shows empty state (no project selected)
2. Status bar shows `○ Spring: not scanned`
3. `lastSpringProjectPath` cleared from settings

## Edge Cases

| Scenario | Behavior |
|----------|----------|
| User selects non-Spring folder | Scan completes with 0 endpoints → show "No endpoints detected" state |
| Project has only `@Controller` (no `@RestController`) | Endpoints still detected and displayed (SPRING-02 covers both) |
| Project has multi-module structure | All modules scanned; endpoints grouped by controller regardless of module origin |
| Project path contains special characters / unicode | Full path displayed as-is; IPC handles encoding |
| Scan takes >10 seconds | Progress bar remains in indeterminate mode; live counter continues updating |
| User closes app during scan | Scan aborted; state not persisted; next launch triggers fresh scan |
| JVM helper crashes during scan | Status bar shows ✗; sidebar shows "Scan failed" error state with error message + Rescan button |
| User clicks endpoint while scan in progress | Click ignored; tooltip shows "Scan in progress — endpoints loading" |
| Same endpoint path with different HTTP methods | Both appear as separate tree entries (e.g., `GET /api/orders` and `POST /api/orders`) |
| Controller with 0 endpoints | Controller not shown in tree (filtered out) |
| Very long controller name (>40 chars) | Truncated with ellipsis; full name in tooltip |
| Very long endpoint path (>60 chars) | Truncated with ellipsis; full path in tooltip |
| DTO class not resolvable | DTO panel shows "No DTO class resolved for this endpoint" |
