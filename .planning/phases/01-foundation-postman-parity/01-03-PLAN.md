## Trust Boundaries (incremental from 01-01 + 01-02)

| Boundary | Description |
|----------|-------------|
| renderer → variables:resolve | Untrusted renderer sends a spec; main resolves it server-side so secrets (env var values) never round-trip through renderer state. Resolved auth credentials stay in main and flow directly to sendRequest. |
| renderer → history:list | Untrusted renderer requests history; main returns the on-disk masked entries (already scrubbed at write time per M-3). Renderer cannot see unmasked auth values via the history IPC. |
| renderer → collections:* | Untrusted renderer reads/writes collections; main re-validates with Zod on every read+write (D-37: structural round-trip). Masked auth in collection.auth is preserved; renderer cannot bypass the mask by reading the raw file. |
| renderer → import-export:importPostman | Untrusted renderer sends a JSON text; main parses + Zod-validates against Postman v2.1 schema (D-36); unparseable input is rejected; a fresh `_postman_id` is generated; masked secrets in the import are preserved (re-importing the same file does not leak real secrets). |
| renderer → import-export:exportPostman | Untrusted renderer requests an export; main reads the in-memory collection (already masked at write time), serializes to v2.1, returns the JSON; the export does not contain real secrets (D-26 / M-3). |
| main → globals.json (read/write) | Untrusted renderer can write global variables via `globals:update`; main Zod-validates each variable (no prototype pollution, no script injection); the globals are merged into the scope chain at resolve time. |
| main → data dir | User-chosen data dir could be inside a cloud-synced folder. The first-run picker (01-01) and Settings → Data location (this plan) detect this and show a warning chip (D-28). Atomic writes prevent partial corruption; redaction in history prevents secret leakage. |
